> **التاريخ:** 2026-06-11
> **المصدر:** `issues_overview_report.csv` (فحص Screaming Frog لـ www.modonty.com) + تحقّق فعلي من HTML الإنتاج الخام + المصادر الرسمية (Google · React 19).
> **نوع التقرير:** مراجعة فقط — بدون أي تعديل كود.

## ⚡ الخلاصة في سطرين

- فيه **نمط جذري واحد**: على ~‎68٪‎ من الصفحات، الـ `title`/`canonical`/`hreflang`/`robots` تطلع **خارج `<head>`** (داخل الـ `body`).
- **بعد التحقّق الرسمي:** خطورته **منخفضة** — Next يخدم الميتا في الـ `<head>` للبوتات المهمة (Google/Bing/واتساب/فيسبوك/تويتر)؛ فقط الزواحف بـ UA عام (زي Screaming Frog) تشوفها في الـ `body`. الإصلاح = تنظيف + أداء، مو طوارئ.
- باقي بنود التقرير: إما **إنذارات كاذبة** (هيدرز الأمان موجودة فعلاً)، أو **تحسينات بسيطة**.

### الدليل الفعلي (بايت من HTML الإنتاج)

صفحة العميل `/clients/شركة-جبر-سيو`:
```
</head>  يُغلق عند بايت   3,193   ← القشرة الساكنة
<title>          عند بايت  157,035  ← داخل الـ body
canonical        عند بايت  157,203  ← داخل الـ body
robots           عند بايت  157,158  ← داخل الـ body
og:title         عند بايت  154,680  ← داخل الـ body
```
وبعد `</head>` مباشرة:  `<div hidden><!--$?--><template id="B:0">` = علامة بثّ Suspense.

### السبب التقني

هالصفحات تُعرض **ديناميكياً** (تقرأ بيانات وقت الطلب — كوكيز/جلسة/`searchParams` أو استعلام DB غير مكاش). مع `cacheComponents`/PPR في Next 16، Next يبثّ الصفحة وتطلع الميتا في الـ `body` عشان React 19 يرفعها للـ `head` **في المتصفح**. المقالات تقرأ بيانات مكاشة (`'use cache'` اللي ضفناه) → ميتاها داخل القشرة الساكنة = داخل الـ `head`. أي زاحف بلا JavaScript (وضع Screaming Frog الافتراضي، وأغلب كاشطات السوشيال) يشوفها في الـ `body`.

### الضرر الحقيقي — مُصحّح بعد التحقّق الرسمي (2026-06-11)

> **تصحيح:** تقديري الأولي «معاينة السوشيال مكسورة» كان **غلط**. وثائق Next 16.3 المرفقة + ملف البوتات الفعلي (`html-bots.ts`) يثبتون إن Next يخدم الميتا في الـ `<head>` (blocking) للبوتات المعروفة، ويبثّها في الـ `body` فقط للزواحف غير المدرجة + المتصفحات.

**قائمة Next الافتراضية اللي تاخذ الميتا في الـ `<head>`:** WhatsApp · facebookexternalhit · facebookcatalog · Twitterbot · LinkedInBot · Slackbot · Bingbot · BingPreview · Discordbot · redditbot · Google · Google-PageRenderer.

| الجهة | الحقيقة |
|---|---|
| واتساب · فيسبوك · تويتر · لينكدإن · سلاك · ديسكورد | ✅ ياخذون الميتا في الـ `<head>` → المعاينة سليمة |
| Google · Bing | ✅ سليمة (blocking أو يعرضون JavaScript) |
| Screaming Frog (UA عام) + زواحف غير مدرجة | ⚠️ يشوفونها في الـ `body` → تحذير شكلي فقط |

### الخطورة: 🟢 منخفضة (بعد التصحيح)

تحذير Screaming Frog صحيح لـ UA الأداة، لكنه **لا يعكس** اللي يستقبله Google/Bing/السوشيال. هذا **مو باج في Next — سلوك مقصود** (streaming metadata كميزة أداء) يتفعّل بسبب قراءاتنا غير المكاشة في `generateMetadata`. **الإصلاح = تنظيف + أداء**، مو طوارئ SEO: تحويل القراءات لمكاشة (`'use cache'` + `env` للـ siteUrl) يسكت التحذير + يحط الميتا في القشرة الساكنة + يشيل قراءات DB غير مكاشة من كل طلب (يساعد مشكلة السعة). وهو حرفياً حل Next الموصى به.

## ⚪ إنذارات كاذبة / بالتصميم — تجاهلها

## ✅ تحقّق بنفسك (يثبّت التشخيص)

شغّل JavaScript Rendering في Screaming Frog:
`Config → Spider → Rendering → JavaScript` ثم أعد الفحص.

→ مجموعة «outside head» المفروض **تختفي**، لأن React 19 يرفع الوسوم للـ `head` بعد تشغيل الـ JavaScript. هذا يثبّت إن المشكلة في **بثّ HTML الأولي**، مو في الكود نفسه.
