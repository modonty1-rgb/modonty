موظف العملاء الكاتب المصمّم النظام تلقائياً

أ) سيو العميل

قبل
موظف العملاء يلقى قدّامه حقول سيو ما يفهمها ولا هي شغله.
بعد
الحقول تنتقل للكاتب. صفحة العميل نظيفة، يبقى فيها شارة النسبة تنبّه لو ناقص.
١
موظف العملاء يدخل بيانات العميل عادي. يشوف بس شارة «SEO ناقص ٦٠٪».
٢
الشارة الناقصة = تنبيه إن العميل يحتاج كاتب يكمّل سيوه.
٣
الكاتب يفتح SEO Client → قائمة العملاء وجنب كلٍّ نسبته.
٤
يختار عميلاً ناقصاً → يشوف بيانات العميل قراءة‑فقط كسياق + يكتب العنوان والوصف.
٥
حفظ → النظام يعيد بناء بيانات قوقل للعميل + كل مقالاته، وينشر لمدوّنتي.
٦
النسبة ترتفع وتظهر محدّثة في كل مكان.

ب) سيو الصور

قبل
المصمّم يُطلب منه سيو ما يفهمه فيتركه ناقص · وتعديل الصورة ما كان يحدّث العميل لو شعاره.
بعد
المصمّم يرفع بس. الكاتب يكتب السيو، والحفظ يحدّث كل مكان تظهر فيه الصورة.
١
المصمّم يرفع الصورة ويقصّها ويصمّمها. ما يشوف حقول سيو.
٢
الكاتب يفتح SEO Images → قائمة الصور وجنب كلٍّ نسبتها.
٣
يختار صورة ناقصة → يشوف الصورة + وين مستخدمة كسياق + يكتب النص البديل · العنوان · الوصف.
٤
حفظ → النظام يحدّث كل مقال يستخدم الصورة، ولو شعار/غلاف عميل يحدّث سيو العميل كمان (الثغرة المُصلَحة)، وينشر.
٥
نسبة الصورة ترتفع وتظهر في القائمة.

🔎 المراجعة العكسية النهائية — ثغرات ما مسكتها الخطة (محلولة)

#الثغرة (من الكود)الحل
🔒 N1 بناء ImageObject متكرّر في مولّدين وغير متسق: مولّد العميل يخرج caption فقط · مولّد المقال يخرج name(=alt)+caption · الاثنان بلا license. (+معاينة الأدمن تبنيه بطريقة ثالثة → تكذب.) دالة مشتركة واحدة buildImageObject() في dataLayer يستدعيها: مولّد العميل + مولّد المقال + معاينة الأدمن. كل صورة بأي مكان تطلع بنفس الحقول (alt/name/description/dimensions + كل الترخيص). مصدر واحد. (يغطّي N3: المعاينة = الحقيقة.)
🔒 N2 بوابة النشر تفحص وجود صورة رئيسية (featuredImageId) بس، مو نصّها البديل → مع alt الاختياري (R3) مقال ممكن يُنشر وصورته بلا alt. نضيف فحص alt الصورة الرئيسية لسكور المقال/بوابة النشر → يتوافق مع «١٠٠٪ قبل الفهرسة». alt الاختياري وقت الرفع لا يسرّب صورة ناقصة للنشر.
🔒 M1 فخّ الـ select — المولّدان غير متماثلين: مولّد المقال featuredImage: true (كل الحقول ✅) · مولّد العميل logoMedia/heroImageMedia: select{url,altText,width,height} = ٤ فقط ❌. بلا توسيع → الترخيص/الاسم يطلعون فاضيين بصمت. عند بناء buildImageObject() نوسّع select الوسائط في مولّد العميل — ثلاثة مصادر: logo · hero · gallery (+ تحويل galleryImages يمرّر الحقول) — ليشمل title/description/credit/creator/license/type/dateCreated/cloudinaryPublicId، يطابق المقال.
🔒 M2 ترقيم الغاليري: اسم «{العميل} — صورة {رقم}» يحتاج ترتيباً ثابتاً، والوسيط بلا حقل ترتيب. نرقّم من ترتيب ثابت (createdAt) أو نشيل الرقم («{العميل} — صورة»).
✅ M3 مبرّأة: شيل حقول السيو من فورم العميل — هل يكسر الإنشاء؟ seoTitle/seoDescription اختياريان في السكيما → آمن (إنشاء + تعديل).

🔴 مخاطر التنفيذ (نقد ذاتي — متحقّق من الكود)

#حدّةالمشكلةالمعالجة قبل البدء
R1✅ محلول نموذج الكاتب = seoTitle + seoDescription فقط. بعد الفحص: Knows Language يجي من الـ settings (وخانته ميّتة أصلاً — تُشال) · GBP (٥ حقول، ميزة غير مفعّلة) + Price Range + الإحداثيات لكل عميل → تبقى بقسم العملاء (أدمن). الحقول المنصّية (robots/twitter/canonical/locale) أصلاً غير معروضة. الكاتب: عنوان+وصف. التقني/المحلي: قسم العملاء. + شيل خانة Knows Language من نموذج العميل.
R2✅ محلول server action واحد: update العميل + إعادة توليد JSON-LD والـ Meta (النمط الموجود updateClient → generateClientSEO). القلق مو في الفكرة — بس التنفيذ: لو أُرسل الكائن كامل بحقول فاضية قد يمسح. update جزئي — يكتب فقط الحقول المُرسَلة (seoTitle/seoDescription) ويتجاهل غير المُرسَل (الكود يسويه أصلاً بالمقارنة)، ثم يعيد التوليد.
R3✅ محلول المصمّم يرفع الصورة بس — بلا أي داتا سيو. نشيل نموذج السيو من شاشة الرفع (upload-zone/seo-form)، وaltText يصير اختياري في createMedia. تحرير alt/title/description كله من قسم المحتوى (الكاتب). الصورة تطلع بـ alt فاضٍ = «ناقصة» بالسكور لين الكاتب يكمّلها (الفلو المقصود). ننبّه: أي عرض يعتمد alt يتحمّل الفراغ مؤقتاً.
🔒 R4مقفول ✅ قائمة عمل الكاتب «الناقص أولاً». تحقّقنا: ما فيه seoScore مخزّن لأي كيان — العميل/المقال/الصور كلهم يحسبون لحظياً وينجح. حساب لحظي في الذاكرة (الخيار أ) — نفس نمط العميل/المقال، صفر عمود جديد وصفر تعارض. التخزين (عمود seoScore) مؤجّل، فقط لو أثبت العدد الكبير بطئاً.
