1)المشكلة والهدف

المشكلة: عندنا الآن زرّان معطّلان (placeholders): «احجز الآن» في الدوك السفلي للجوال، و«احجز أونلاين قريباً» في كارت العميل. والفكرة الأصلية تفترض إن كل عميل عنده «مواعيد» — بينما جمهور مُدوّنتي عريض (دكتور · محامي · مستشار · متجر · تاجر). فرض «احجز موعد» على متجر إلكتروني = بلا معنى.

الهدف: زرّ إجراء رئيسي واحد، قابل للتهيئة لكل عميل، يظهر في كل مكان يظهر فيه العميل (المقال + الكارت + صفحة العميل لاحقاً)، ويوصّل lead فوري للعميل عبر القنوات الموجودة عندنا أصلاً.

2)القرار الجوهري 🔒 مقفول ✅ وافق خالد 2026-06-05

الزرّ مو ثابت — هو نسخة من «إجراء رئيسي» يتكوّن من:

الإجراء = النوع (mode) + التسمية (label) + الوجهة (target)

نرتكز على نوع الإجراء، مو على المهنة — لأن مهنتين متطابقتين قد يبغون إجراءين مختلفين، والربط بالمهنة يفرض جدول صيانة بلا فايدة.

4.2 — موديل الحجوزات (لو اخترنا dedicated — ❓2)

model BookingRequest {
  id          String   @id @default(auto()) @map("_id") @db.ObjectId
  clientId    String   @db.ObjectId
  client      Client   @relation(...)
  articleId   String?  @db.ObjectId   // null لو الحجز من صفحة العميل/القائمة (مو من مقال)
  userId      String?  @db.ObjectId   // SetNull عند حذف الحساب — الـ snapshot يحفظ الـ lead
  source      String                  // article_dock | article_card | client_page | client_list
  name        String                  // snapshot من الجلسة
  email       String                  // snapshot من الجلسة
  phone       String                  // يجمعه الفورم (User ما فيه جوال)
  preferredAt DateTime?               // الموعد المفضّل (اختياري، مستقبلي)
  message     String?
  status      String   @default("new")  // new|contacted|done|archived
  ipAddress   String?
  userAgent   String?
  createdAt   DateTime @default(now())
  @@index([clientId, createdAt])
  @@index([userId])
  @@map("booking_requests")
}

5)الأسطح + تحليل UX ✅ خالد 2026-06-05

🐞 مشكلة UX في دوك الجوال (نصلحها): الزر المركزي مكتوب «احجز الآن» لكن ضغطه يفتح كارت العميل مو الحجز → (١) عدم تطابق التسمية مع الفعل، (٢) نقرتان للحجز (تفتح الكارت ثم تدوّر الزر جواه). احتكاك يقتل التحويل.

الحل — «شيت كامل يبدأ بالحجز» (booking-first): ضغطة واحدة على شعار العميل تفتح شيت طويل ~90vh (كأنه صفحة جوال مخصّصة، تحكّم كامل) — عبر مكوّن Sheet الحالي بدون مكتبة جديدة (حماية bundle مودونتي؛ vaul/Drawer غير مثبّت ولا يستاهل يُضاف).

الترتيب من فوق لتحت (في نفس الشيت — بدون drawer ثانٍ):

• ١. بيانات كارت العميل الكامل — نعيد استخدام الكارت الموجود ArticleLabClientCard نفسه (DRY): شعار/هيرو + اسم + ✓موثّق + المدينة + نبذة + تواصل/سوشال.

• ٢. الفورم — جوال (إلزامي) + موعد مفضّل (اختياري) + رسالة (اختياري). [للـ FORM فقط]

• ٣. Footer ثابت — زر «تأكيد الحجز» = الـ CTA الواضح (submit الفورم). للـ LINK: نفس الكارت بدون فورم، والفوتر = زر «تسوّق الآن».

تمرير داخلي للمحتوى · الفوتر ثابت دايماً ظاهر · إغلاق: X + سحب لتحت + ضغط الـ 10% الفاضية فوق + Escape (إتاحة، تركيز محبوس).

حالات تسجيل الدخول (غطّينا الأربعة): كارت العميل يظهر دايماً (مسجّل أو لا).

6)التدفّقات

🗂️ mode = FORM تسجيل دخول إلزامي

٠. غير المسجّل → «سجّل الدخول للحجز». المسجّل → يفتح الفورم.

١. الفورم: الجوال (إلزامي) · موعد مفضّل (اختياري) · رسالة. الاسم/الإيميل من الجلسة، الدولة من IP.

٢. Submit → BookingRequest + Notification + notifyTelegram + Conversion + CTAClick + GA4.

٣. العميل يشوف: Telegram خاص + إشعار داخلي + الحجز بصندوق الحجوزات (❓6).

🔗 mode = LINK تسجيل دخول إلزامي

٠. غير المسجّل → «سجّل الدخول». المسجّل → يكمّل.

١. ضغط الزر → CTAClick + Conversion (مربوطة بـ userId = lead دافئ) + GA4.

٢. فتح ctaUrl مباشرة في تبويب جديد (rel="noopener") — بدون شيت. نعيد استخدام CtaTrackedLink.

🔗 ربط الإجراء بحساب الزائر (مهم): كل إجراء (فورم + رابط) مربوط بـ userId تلقائياً (إلزامي بالموديل) — بدون تعديل سكيما إضافي. ثلاث فوائد:

١. الزائر يشوف «حجوزاتي/نشاطي» ببروفايله + حالة كل حجز. (عرض — Phase 5)

٢. العميل يشوف الـ leads الواصلة له. (عرض — Phase 5)

٣. مستقبلاً: نستخرج «العملاء الحقيقيين» = الزوار عالي النية → فرص تحويل. (تحليلات مستقبلية)

ج. إقرار المسؤولية عن المحتوى — شامل (كل ما يسلّمه العميل)

يغطّي: بيانات · صور · وثائق · تراخيص · فيديوهات — أي شي يسلّمه العميل = على مسؤوليته الكاملة. مُدوّنتي تنشر فقط بلا أي مسؤولية.

• الأدمن: صفحة تحرير نص الإقرار الكامل + قيمة افتراضية (Default) — قابل للتعديل مركزياً.

• الكونسول: النص يظهر بارز للعميل + checkbox إلزامي قبل الحفظ/الموافقة.

• تسجيل القبول: مين (userId) + متى (acceptedAt) + نسخة النص (version) — وإلا الإقرار بلا قيمة إثباتية.

• ⚠️ تنبيه: محامي يراجع النص النهائي قبل الإنتاج (YMYL حسّاس قانونياً بالسعودية/مصر). الصياغة أدناه عملية، مو فتوى قانونية.

النص الافتراضي المقترح (قابل للتعديل بالأدمن):

«كل ما أزوّد به مُدوّنتي — من بيانات وصور ووثائق وتراخيص وفيديوهات — صحيح وملكي وأملك حق نشره، وأنا المسؤول الوحيد عنه قانونياً. دور مُدوّنتي هو النشر فقط، وهي غير مسؤولة عن صحّته أو مصدره أو قانونيّته.»

• «حجوزاتي» للزائر: تبويب جديد /users/profile/bookings (الـ hub موجود).

المساران في هذا المشروع:

• Track B (visitor): زر الحجز على المقال + صفحة العميل + القائمة + الشيت — تميّز بصري ضمن قيود الـ stack، أداء + bundle حسّاس على مودونتي (أصغر client boundary).

• Track A (admin/console): فورم الأدمن + صندوق الحجوزات بالكونسول — وضوح + هرمية + progressive disclosure + بؤرة مهيمنة.

الطقس الإلزامي قبل أي UI بنيوي: brief (٢٠ كلمة) ← هرمية ← mockup HTML أول ← مراجعتك ← كود مرّة وحدة. الشيت (90vh) + صندوق الحجوزات = بنيوية → mockup أول قبل الكود.

التحقّق قبل الـ push: live test (Playwright) مكتب + جوال RTL · صفر overflow · a11y (aria على الأزرار الأيقونية + focus trap بالشيت) · صفر console errors.

ملاحظة صادقة: ما أقدر «أنزّل» skills خارجية عشوائية من النت، لكن قواعد Vercel/Anthropic/WCAG الصناعية مدمجة أصلاً في skill modonty-uiux المعتمد — وربطتها هنا عشان تكون مُلزِمة لهذا البناء.
